3.详细介绍

playbook是由一个或多个play组成的列表
play的主要功能在将实现归并为一组的主机装扮成事先通过ansible中的task定义好的角色。从根本上来讲，所谓task无非是调用ansible的一个module。
将多个play组织在一个playbook，就可以让他们联同起来按实现编排的机制完成任务

3.1 playbook核心元素

  • 1.hosts

    执行的远程主机列表

  • 2.tasks

    任务集

  • 3.varniables

    内置变量或自定义变量在playbook中调用

  • 4.templates

    模板，可替换模板文件中的变量并实现一些简单逻辑的文件

  • 5.Handlers

    和notity结合使用，由特定条件触发的操作，满足条件方才执行，否则不执行

  • 6.tags
    标签，指定某条任务执行，用于选择运行playbook中的部分代码。
    ansible具有幂等性，因此会自动跳过没有变化的部分，即便如此，有些代码为测试其确实没有发生变化的时间依然会非常地长。
    此时，如果确信其没有变化，就可以通过tags跳过这些代码片段

3.2 playbook基础组件

3.2.1 hosts

用于指定要执行指定任务的主机
可以有如下格式:
192.168.1.1
192.168.1.*
group01.webser.com
group01.webser.com:group02.webser.com
webser:dbser # 两个组的并集
webser:&dbser # 两个组的交集
webser:!dbser # 在webser组但不在dbser组

  • 示例

3.2.2 remote_user

可用于hosts和tasks中。
也可以通过指定通过sudo的方式在远处上执行任务，其可用play全局或某任务；此外，甚至可以在sudo时使用sudo_user指定sudo切换的用户
可以有如下格式
remote_user: test1
sudo: yes # 默认sudo用户为root
sudo_user: admin # 指定sudo用户为admin

3.3 handlers和notify结合使用触发条件

  • 3.3.1 handlers

    是task列表，这些task与之前说的task并没有本质上的不同，用于当关注的资源发生变化时，才会采取一定的操作

  • 3.3.2 notify

    此action可用于在每个play的最后被触发，
    这样可避免多次有改变发生时都执行指定的操作，仅在所有的变化发生完成后一次性地执行指定操作。
    在notify中列出的操作成为handler，也就是notify中调用handler中定义的操作

3.5 变量的使用

变量名：仅能由字母、数字和下划线组成，且只能以字母开头
变量来源：
1、ansible setup facts 远程主机的所有变量都可直接调用
2、在/etc/ansible/hosts/中定义
普通变量：主机组中主机单独定义，优先级高于公共变量
公共（组变量):针对主机组中所有主机定义统一变量
3、通过命令行指定变量，优先级最高
ansible-playbook -e varname=value
4、在playbook中定义
vars:
- var1: value1
- var2: value2

6.vars_propmt交互式定义变量
